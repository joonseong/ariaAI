# PRD: Aria — AI 그림 작가를 위한 포트폴리오 & 커뮤니티 앱

> 버전: 0.3 (MVP)
> 작성일: 2026-04-23
> 플랫폼: React Native (iOS / Android)

---

## 1. 목표

AI 그림 작가들이 작품을 손쉽게 공유하고, 서로 팔로우·소통하며, 자신만의 포트폴리오를 쌓아나갈 수 있는 모바일 커뮤니티를 만든다.

**해결하는 문제**
- AI 작가들은 작품을 올릴 전용 공간이 없어 SNS에 파편화되어 있다.
- 포트폴리오 관리, 팬과의 소통, 작품 검색이 한 곳에서 이뤄지지 않는다.

**해결하지 않는 문제 (명시적 비목표)**
- AI 이미지 생성 도구를 앱 안에서 제공하지 않는다. Aria는 "갤러리"이지 "도구"가 아니다.
- 작품의 상업적 거래(판매, 라이선스)는 MVP 범위 밖이다.
- AI vs 인간 작품의 구분·인증 시스템은 제공하지 않는다.

---

## 2. 사용자 (타겟)

| 구분 | 설명 | 핵심 니즈 |
|------|------|-----------|
| **주 사용자 — 크리에이터** | AI 이미지 생성 도구(Midjourney, DALL-E, Stable Diffusion, ComfyUI 등)를 사용하는 작가 | 작품 아카이빙, 팬 확보, 포트폴리오 구축 |
| **부 사용자 — 감상자** | AI 아트를 감상하고 작가를 팔로우하고 싶은 일반 사용자 | 영감 탐색, 좋아하는 작가 발견, 소통 |
| **비사용자 — 비회원** | 앱을 설치했지만 가입하지 않은 탐색자 | 가입 전 콘텐츠 맛보기 |

### 사용자 페르소나

**페르소나 A — 활동적 크리에이터 "민지"**
- 25세, Midjourney·Stable Diffusion으로 매일 3~5장 생성
- 현재 Twitter/Instagram에 올리지만 AI 아트에 특화된 피드백이 없음
- 원하는 것: 작품을 체계적으로 정리하고, 비슷한 취향의 팬을 모으고 싶음

**페르소나 B — 캐주얼 감상자 "준혁"**
- 30세, AI 아트를 좋아하지만 직접 만들지는 않음
- Pinterest에서 AI 아트를 모으지만, 작가를 팔로우하기가 어려움
- 원하는 것: 좋아하는 작가를 팔로우하고 새 작품을 놓치지 않고 싶음

## 3. 핵심 기능 (MVP 범위)

### 3-1. 인증 (로그인 / 회원가입)

#### 기본 동작
- 이메일 + 비밀번호 회원가입
- 이메일 + 비밀번호 로그인
- 소셜 로그인: Google, Apple
- Firebase Auth의 ID 토큰 기반 세션 유지 (자동 갱신)
- 자동 로그인: 앱 재시작 시 이전 세션 복원

#### 회원가입 플로우
```
1. 이메일 입력 → 형식 검증 (RFC 5322)
2. 비밀번호 입력 → 강도 검증
3. 닉네임 입력 → 중복 검사
4. 약관 동의 (필수: 이용약관, 개인정보처리방침)
5. Firebase Auth createUserWithEmailAndPassword()
6. Firestore users/{uid} 문서 생성
7. 홈 피드로 이동
```

#### 입력 검증 규칙
| 필드 | 규칙 | 에러 메시지 |
|------|------|-------------|
| 이메일 | RFC 5322 형식, 최대 254자 | "올바른 이메일 주소를 입력해주세요" |
| 비밀번호 | 최소 8자, 영문+숫자 필수 | "비밀번호는 영문과 숫자를 포함하여 8자 이상이어야 합니다" |
| 닉네임 | 2~20자, 한글/영문/숫자/밑줄만 허용, 공백 불가 | "닉네임은 2~20자, 한글·영문·숫자·밑줄만 사용 가능합니다" |
| 닉네임 중복 | Firestore에서 실시간 중복 체크 | "이미 사용 중인 닉네임입니다" |

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 이미 가입된 이메일 | "이미 가입된 이메일입니다. 로그인해주세요" + 로그인 화면 링크 |
| 소셜 로그인 취소 | 로그인 화면으로 복귀, 토스트 없음 (사용자 의도적 취소) |
| 소셜 로그인 실패 (네트워크) | "네트워크 연결을 확인해주세요" 토스트 |
| 비밀번호 틀림 | "이메일 또는 비밀번호가 올바르지 않습니다" (보안상 어느 쪽인지 구분하지 않음) |
| 5회 연속 로그인 실패 | Firebase Auth 자동 잠금 (일정 시간 후 해제) |
| 탈퇴한 계정으로 재로그인 | "존재하지 않는 계정입니다" (탈퇴 후 데이터 삭제 완료 상태) |
| Apple 로그인 — 이메일 숨기기 선택 | Apple의 프라이빗 릴레이 이메일 사용, 정상 가입 처리 |
| 토큰 만료 | Firebase SDK가 자동 갱신. 갱신 실패 시 로그인 화면으로 이동 + "세션이 만료되었습니다. 다시 로그인해주세요" |

#### 비밀번호 재설정
```
1. 로그인 화면 > "비밀번호를 잊으셨나요?" 탭
2. 이메일 입력
3. Firebase Auth sendPasswordResetEmail()
4. "비밀번호 재설정 메일을 발송했습니다" 토스트
5. 미가입 이메일이어도 동일한 메시지 표시 (이메일 존재 여부 노출 방지)
```

> **초보자 참고 — 세션과 토큰**
> "로그인 상태를 유지한다"는 것은 앱이 "이 사용자가 누구인지" 증명하는 토큰(일종의 출입증)을 가지고 있다는 의미입니다.
> Firebase Auth는 이 토큰을 자동으로 관리해주므로, 개발자가 직접 토큰을 저장·갱신할 필요가 없습니다.
> 앱을 껐다 켜도 토큰이 유효하면 자동 로그인됩니다.

### 3-2. 홈 피드

#### 기본 동작
- 최신순 무한 스크롤 피드
- 각 카드: 작품 썸네일(4:3), 제목(1줄, 최대 20자 말줄임), 작가 아바타+닉네임, 좋아요 수, 등록 시간(상대 표기)
- Pull-to-refresh로 최신 데이터 갱신
- 비회원 접근 가능: 홈 피드 열람, 작품 상세 보기, 작가 포트폴리오 보기, 검색
- 비회원 접근 불가 (로그인 유도): 좋아요, 팔로우, 방명록 작성, 작품 등록, 저장
- 비회원 가입 전환 유도: 좋아요/팔로우 시도 시 바텀시트에 "가입하면 좋아하는 작가의 새 작품을 가장 먼저 만나보세요" 문구 + 가입 CTA 버튼

#### 피드 알고리즘 (MVP)
```
1순위: 팔로우 작가 신작 (최근 48시간, 시간순)
2순위: 팔로우하지 않은 작가의 인기 작품 ("추천 작품" 라벨 표시)
3순위: 신규 작가 작품 (가입 7일 이내 작가, "새로운 작가" 라벨 표시)
비회원/신규 가입: 전체 인기순 (최근 7일 좋아요 수 기준)
```
> 이렇게 하면 팔로우하지 않은 다양한 작가를 자연스럽게 발견할 수 있다.

#### 페이지네이션
- Firestore `startAfter()` 커서 기반 페이지네이션
- 페이지당 20개 작품
- 마지막 문서의 `createdAt`을 커서로 사용
- 더 이상 데이터가 없으면 "모든 작품을 확인했습니다" 메시지 표시

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 초기 로딩 실패 (네트워크) | 전체 화면 에러 상태: "피드를 불러올 수 없습니다" + "다시 시도" 버튼 |
| 추가 로딩 실패 (무한 스크롤 중) | 리스트 하단에 "불러오기 실패. 탭하여 재시도" 표시 |
| 빈 피드 (팔로우 0명 + 작품 0개) | 빈 상태 일러스트 + "아직 작품이 없습니다. 작가를 팔로우해보세요!" + 추천 작가 카드 3개 |
| 팔로우한 작가가 있지만 작품이 없음 | 인기 작품 추천으로 대체 |
| 삭제된 작품이 피드에 노출 | 카드 탭 시 "삭제된 작품입니다" 토스트 + 피드에서 해당 카드 제거 |
| 차단/신고된 작가의 작품 | 피드에서 자동 필터링 (내가 신고한 작가의 작품은 표시하지 않음) |

#### 엣지 케이스
| 상황 | 처리 |
|------|------|
| 스크롤 중 네트워크 끊김 | 이미 로딩된 카드는 유지, 추가 로딩 중단, 네트워크 복원 시 자동 재시도 |
| 동일 작품 중복 로딩 (커서 문제) | artworkId 기반 중복 제거 |
| 작가가 탈퇴한 경우 | 해당 작가 작품은 피드에서 숨김 |
| 매우 긴 제목 | 2줄까지 표시, 이후 말줄임(...) |

### 3-3. 작품 상세

#### 기본 동작
- 고해상도 이미지 전체 보기 (핀치 줌 지원)
- 이미지가 여러 장인 경우 좌우 스와이프 (도트 인디케이터)
- 작품 메타데이터: 제목, 설명, 사용 도구 태그, 등록일
- 좋아요 버튼 (하트 토글, 바운스 애니메이션)
- 작가 미니 카드 (아바타 + 닉네임 + 팔로우 버튼)
- 공유 버튼 (시스템 Share Sheet)
- 신고 버튼 (더보기 메뉴 내)

#### 이미지 뷰어 동작
| 제스처 | 동작 |
|--------|------|
| 탭 | UI 오버레이 토글 (제목, 버튼 등 숨기기/보이기) |
| 더블 탭 | 2x 줌 토글 |
| 핀치 | 줌 인/아웃 (최소 1x, 최대 4x) |
| 좌우 스와이프 | 다음/이전 이미지 (여러 장일 때) |
| 아래로 스와이프 | 상세 화면 닫기 (이전 화면으로) |

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 작품 ID가 유효하지 않음 (잘못된 딥링크) | "작품을 찾을 수 없습니다" 전체 화면 + 뒤로가기 버튼 |
| 이미지 로딩 실패 | 이미지 영역에 깨진 이미지 아이콘 + "이미지를 불러올 수 없습니다" + 재시도 버튼 |
| 작가가 작품을 삭제한 상태 | "삭제된 작품입니다" 전체 화면 + 2초 후 자동으로 이전 화면 이동 |
| 좋아요 요청 실패 | 하트를 원래 상태로 되돌림 (낙관적 업데이트 롤백) + "잠시 후 다시 시도해주세요" 토스트 |
| 비회원이 좋아요 탭 | 로그인 유도 바텀시트: "좋아요를 누르려면 로그인이 필요합니다" + 로그인/가입 버튼 |
| 공유 실패 | 시스템 Share Sheet의 에러 처리에 위임 (앱에서 별도 처리 없음) |

### 3-4. 작품 등록

#### 기본 동작
- 갤러리에서 이미지 선택 (최대 5장)
- 카메라 촬영은 지원하지 않음 (AI 생성 이미지를 갤러리에서 선택하는 유스케이스)
- 제목 입력 (필수)
- 설명 입력 (선택)
- 사용 도구 태그 선택 (프리셋 + 커스텀)
- 업로드 진행률 표시
- 등록 완료 후 작품 상세 화면으로 이동

#### 입력 검증 규칙
| 필드 | 규칙 | 에러 메시지 |
|------|------|-------------|
| 이미지 | 최소 1장, 최대 5장 | "이미지를 1장 이상 선택해주세요" / "이미지는 최대 5장까지 가능합니다" |
| 이미지 크기 | 장당 최대 10MB | "이미지 크기는 10MB 이하여야 합니다" |
| 이미지 형식 | JPEG, PNG, WebP만 허용 | "지원하지 않는 이미지 형식입니다 (JPEG, PNG, WebP만 가능)" |
| 이미지 최소 해상도 | 300x300px 이상 | "이미지 해상도가 너무 낮습니다 (최소 300x300)" |
| 제목 | 1~100자, 공백만으로 이루어질 수 없음 | "제목을 입력해주세요" / "제목은 100자 이내로 입력해주세요" |
| 설명 | 최대 2000자 (선택) | "설명은 2000자 이내로 입력해주세요" |
| 태그 | 최대 10개, 각 태그 최대 30자 | "태그는 최대 10개까지 가능합니다" |

#### 프리셋 도구 태그
`Midjourney`, `DALL-E`, `Stable Diffusion`, `ComfyUI`, `Leonardo AI`, `Adobe Firefly`, `기타`

#### 업로드 플로우
```
1. 이미지 선택 → 클라이언트에서 리사이징 (긴 변 기준 최대 2048px)
2. 제목·설명·태그 입력
3. "등록" 버튼 탭
4. 버튼 비활성화 + 프로그레스 바 표시
5. 이미지 1장씩 순차 업로드 → Firebase Storage
   - 각 이미지 업로드 완료 시 진행률 업데이트 (1/5, 2/5...)
6. 모든 이미지 URL 확보 후 Firestore 작품 문서 생성
7. 성공 → 작품 상세 화면으로 이동 + "작품이 등록되었습니다" 토스트
```

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 갤러리 접근 권한 거부 | "사진 접근 권한이 필요합니다" + 설정 열기 버튼 |
| 업로드 중 네트워크 끊김 | 업로드 중단, "네트워크 연결이 끊겼습니다. 다시 시도해주세요" + 재시도 버튼. 이미 업로드된 이미지는 유지하고 남은 것만 재시도 |
| 업로드 중 앱 백그라운드 전환 | 업로드 계속 진행 (iOS 백그라운드 태스크, 최대 30초). 30초 초과 시 중단 → 포그라운드 복귀 시 재시도 안내 |
| 이미지 파일이 손상됨 | "이미지를 처리할 수 없습니다. 다른 이미지를 선택해주세요" |
| Firestore 문서 생성 실패 (이미지는 업로드 완료) | Storage에 업로드된 이미지 URL은 보존, Firestore 재시도. 3회 실패 시 "등록에 실패했습니다. 잠시 후 다시 시도해주세요" + 등록 화면 유지 (입력 데이터 보존) |
| 동시에 여러 번 등록 버튼 탭 | 첫 번째 탭만 처리 (버튼 즉시 비활성화) |
| Storage 용량 초과 (Firebase 무료 플랜) | "서버 저장 공간이 부족합니다" — 운영자 알림 필요 |

#### 엣지 케이스
| 상황 | 처리 |
|------|------|
| HEIC 형식 이미지 (iOS) | expo-image-picker가 자동으로 JPEG 변환 |
| GIF 파일 선택 | 첫 프레임만 정적 이미지로 저장 (애니메이션 미지원) |
| 매우 큰 이미지 (4K+, 20MB+) | 클라이언트에서 리사이징 후 10MB 이하로 압축 |
| 이미지 순서 변경 | 드래그 앤 드롭으로 순서 변경 가능. 첫 번째 이미지가 피드 썸네일 |
| 같은 이미지 중복 선택 | 파일 URI 기반으로 중복 감지, "이미 선택된 이미지입니다" 토스트 |

> **초보자 참고 — 이미지 저장**
> 이미지 파일은 앱 DB에 직접 저장하지 않습니다.
> **Firebase Storage** (클라우드 파일 저장소)에 업로드 → URL을 DB에 저장하는 방식입니다.
> 덕분에 DB는 가볍고, 이미지는 CDN으로 빠르게 전송됩니다.

### 3-5. 작가 포트폴리오

#### 기본 동작
- 헤더: 프로필 사진(80px), 닉네임, 한 줄 소개
- 통계: 작품 수 / 팔로워 수 / 팔로잉 수
- 팔로우/언팔로우 버튼 (본인 프로필에서는 "프로필 수정" 버튼으로 대체)
- 탭 전환: 작품 | 방명록
- 작품 탭: 3열 그리드 (1:1 비율), 탭하면 작품 상세로 이동
- 작품 그리드 정렬: 최신순 (변경 불가, MVP)

#### 크리에이터 통계
- 프로필 탭 > "내 통계" 메뉴 (크리에이터 본인만 볼 수 있음, 작품이 1개 이상인 사용자에게만 표시)
- 표시 항목: 총 작품 조회수, 이번 주 좋아요 수, 팔로워 변화 추이 (최근 7일)
- MVP 구현 방식: `analytics/{userId}/daily/{YYYY-MM-DD}` 컬렉션에 일별 스냅샷 저장 → 앱 진입 시 최근 7일분 조회

> **초보자 참고** — "스냅샷"이란?
> 매일 자정에 그날의 조회수, 좋아요 수, 팔로워 수를 기록해두는 것입니다.
> MVP에서는 앱이 열릴 때 클라이언트에서 현재 값을 계산하여 저장합니다.
> (v2에서는 Cloud Function으로 자동화)

#### 방명록 (Guestbook)
- 방문자가 짧은 메시지 남기기
- 작가가 메시지에 답글 달기 (1 depth만, 대대글 없음)
- 시간순 정렬 (최신이 위)
- 페이지네이션: 20개씩 로딩

| 필드 | 규칙 |
|------|------|
| 메시지 | 1~200자, 공백만 불가 |
| 답글 | 1~200자, 작가 본인만 작성 가능 |
| 삭제 | 메시지 작성자 본인 또는 포트폴리오 주인(작가)이 삭제 가능 |

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 존재하지 않는 작가 ID | "작가를 찾을 수 없습니다" 전체 화면 |
| 탈퇴한 작가 | "탈퇴한 작가입니다" 전체 화면 |
| 팔로우 요청 실패 | 버튼 원래 상태로 복원 + "잠시 후 다시 시도해주세요" 토스트 |
| 자기 자신을 팔로우 | UI에서 팔로우 버튼 자체를 숨김 (불가능한 상태를 노출하지 않음) |
| 방명록 작성 실패 | 입력 내용 유지 + "메시지 전송에 실패했습니다" 토스트 |
| 비회원이 방명록 작성 시도 | 로그인 유도 바텀시트 |
| 방명록 스팸 | 동일 사용자가 같은 작가에게 1분 내 3회 이상 작성 시 "잠시 후 다시 작성해주세요" |
| 작품 0개인 작가 | 작품 탭에 빈 상태: "아직 등록된 작품이 없습니다" |

### 3-6. 팔로우 시스템

#### 기본 동작
- 팔로우 → 홈 피드에 해당 작가 작품 노출
- 팔로워/팔로잉 목록 확인 (각각 별도 화면)
- 팔로워/팔로잉 목록에서 직접 팔로우/언팔로우 가능
- **v2 예정**: 팔로우한 작가가 새 작품을 등록하면 팔로워에게 푸시 알림 전송 (ARCHITECTURE.md 푸시 알림 아키텍처 참조)

#### 낙관적 업데이트
```
1. 팔로우 버튼 탭 → 즉시 UI 변경 (버튼 "팔로잉"으로 전환, 팔로워 수 +1)
2. 백그라운드에서 Firestore 트랜잭션 실행
   - follows 문서 생성
   - 대상 유저 followersCount +1
   - 내 followingCount +1
3. 실패 시 → UI 롤백 (버튼 "팔로우"로 복원, 팔로워 수 -1) + 토스트
```

#### 에러/엣지 케이스
| 상황 | 처리 |
|------|------|
| 빠르게 팔로우/언팔로우 반복 (더블 탭) | 디바운싱 300ms. 마지막 상태만 서버에 반영 |
| 팔로우 중 네트워크 끊김 | 낙관적 업데이트 롤백 + 오프라인 토스트 |
| 팔로워 수 음수 방지 | Firestore 트랜잭션에서 `Math.max(0, current - 1)` |
| 팔로우한 작가가 탈퇴 | 해당 follow 문서 자동 정리 (Cloud Function 또는 앱에서 렌더링 시 필터링) |
| 팔로워/팔로잉 목록이 비어있음 | "아직 팔로워가 없습니다" / "아직 팔로우하는 작가가 없습니다" |

### 3-8. 검색

#### 기본 동작
- 검색 탭 진입 시: 인기 태그 표시 + 최근 검색어
- 키워드 입력 → 작품(제목+태그) 검색 + 작가(닉네임) 검색
- 탭으로 "작품" / "작가" 결과 전환
- 검색 결과: 작품은 2열 그리드, 작가는 리스트

#### 검색 구현 (Firestore 제약 고려)
```
MVP 방식: Firestore의 문자열 prefix 매칭
- 작품 제목: where('title', '>=', query).where('title', '<=', query + '\uf8ff')
- 작가 닉네임: 동일 방식
- 태그: where('tags', 'array-contains', tag)
```

> **초보자 참고 — prefix 매칭이란?**
> Firestore는 구글 검색 같은 "전문 검색"을 지원하지 않습니다.
> 대신 "앞글자부터 일치하는 것"만 찾을 수 있습니다. 이것을 prefix 매칭이라고 합니다.
>
> ```
> 검색어: "환상"
>   ✅ "환상의 숲" → 찾음 (앞에서부터 일치)
>   ❌ "몽환상" → 못 찾음 (중간 글자)
>   ❌ "숲" → 못 찾음 (뒤 글자)
>   ❌ "환상 숲" → 못 찾음 (띄어쓰기가 다름)
>   ❌ "환상에 숲" → 못 찾음 (오타)
> ```
>
> 이 한계는 MVP에서는 감수합니다. 사용자가 불편을 느낄 수 있으므로,
> 검색 결과 0건일 때 "앞부분을 입력하면 더 정확한 결과를 얻을 수 있습니다" 힌트를 표시합니다.

#### v2 검색 개선 로드맵 — Algolia

> **Algolia란?**
> 검색 전용 외부 서비스입니다. 구글 검색처럼 중간 글자, 오타, 유사어, 한국어 형태소까지 처리해줍니다.
> Firestore의 데이터를 Algolia에 복사(동기화)해두면, 검색할 때 Algolia에 물어보는 방식입니다.
>
> ```
> [사용자 검색] → Algolia API → 결과 반환 (매우 빠름, <50ms)
>                    ↑
>              [Cloud Function이 자동 동기화]
>                    ↑
>              [Firestore에 작품 생성/수정/삭제]
> ```

| 항목 | 내용 |
|------|------|
| 필요한 것 | Algolia 계정 + API 키 (ALGOLIA_APP_ID, ALGOLIA_API_KEY) |
| 비용 | 무료 한도: 월 10,000건 검색. 유료: 월 $29~ |
| 전제 조건 | Firebase Blaze 플랜 (Cloud Functions 필요) |
| 코드 변경 범위 | `services/search.ts`만 교체 (훅·화면 변경 불필요, ADR-011 참조) |
| MVP에서의 준비 | 검색 로직을 `services/search.ts`에 격리 → v2에서 교체 용이하도록 설계 |

#### 에러/엣지 케이스
| 상황 | 처리 |
|------|------|
| 검색 결과 0건 | "검색 결과가 없습니다" + "작품이나 작가의 앞부분을 입력해보세요" (prefix 한계 안내) |
| 빈 검색어로 검색 시도 | 검색 실행하지 않음 (버튼 비활성) |
| 특수문자만 입력 | 검색 실행하되, 결과 없으면 0건 처리와 동일 |
| 검색어 최대 길이 | 50자 제한, 초과 시 입력 불가 |
| 최근 검색어 개수 | 최대 20개, FIFO로 오래된 것 자동 삭제 |
| 최근 검색어 삭제 | 개별 삭제 (X 버튼) + 전체 삭제 |
| 인기 태그 로딩 실패 | 인기 태그 섹션 숨김, 검색 바만 표시 |
| 빠른 타이핑 (연속 입력) | 디바운싱 500ms. 마지막 입력 기준으로 검색 |
| 네트워크 오프라인 | "검색하려면 인터넷 연결이 필요합니다" |

### 3-9. 프로필 설정

#### 기본 동작
- 프로필 사진 변경 (갤러리에서 선택 + 원형 크롭)
- 닉네임 수정 (중복 검사)
- 한 줄 소개 수정
- 계정 정보 표시 (이메일, 로그인 방식)
- 로그아웃
- 회원 탈퇴

#### 입력 검증 규칙
| 필드 | 규칙 |
|------|------|
| 프로필 사진 | JPEG/PNG, 최대 5MB, 최소 100x100px |
| 닉네임 | 회원가입과 동일 규칙 |
| 한 줄 소개 | 최대 150자 |

#### 회원 탈퇴 플로우
```
1. "회원 탈퇴" 탭
2. 확인 다이얼로그: "정말 탈퇴하시겠습니까? 모든 데이터가 삭제되며 복구할 수 없습니다."
3. 비밀번호 재입력 (소셜 로그인은 재인증)
4. 데이터 삭제 순서:
   a. Firestore: 내 작품 전체 삭제 (artworks where authorId == uid)
   b. Firestore: 내 좋아요 전체 삭제
   c. Firestore: 내 팔로우 관계 전체 삭제 + 상대방 팔로워 수 차감
   d. Firestore: 내 방명록 메시지 전체 삭제
   e. Firebase Storage: 내 이미지 전체 삭제
   f. Firestore: users/{uid} 문서 삭제
   g. Firebase Auth: 계정 삭제
5. 로그인 화면으로 이동 + "탈퇴가 완료되었습니다" 토스트
```

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 닉네임 변경 시 이미 사용 중 | "이미 사용 중인 닉네임입니다" |
| 프로필 사진 업로드 실패 | 기존 사진 유지 + "사진 변경에 실패했습니다" 토스트 |
| 탈퇴 중 일부 데이터 삭제 실패 | 전체 롤백하지 않고, 남은 데이터를 Cloud Function에서 비동기 정리 (MVP에서는 수동 정리) |
| 탈퇴 후 동일 이메일로 재가입 | 새로운 계정으로 취급 (uid 새로 발급) |
| 소셜 로그인 계정 탈퇴 | Firebase Auth 재인증 후 탈퇴 처리 |
| 로그아웃 실패 | 로컬 토큰만이라도 삭제하고 로그인 화면으로 이동 |

### 3-10. 신고 시스템

#### 신고 대상
- 작품 신고
- 방명록 메시지 신고
- 작가 신고

#### 신고 플로우
```
1. 더보기(⋯) 메뉴 > "신고" 선택
2. 신고 사유 선택 (바텀시트):
   - 스팸/광고
   - 불쾌한 콘텐츠
   - 저작권 침해
   - 기타 (직접 입력, 최대 500자)
3. "신고" 버튼 탭
4. Firestore reports 컬렉션에 신고 문서 생성
5. "신고가 접수되었습니다" 토스트
6. 같은 콘텐츠 재신고 불가 ("이미 신고한 콘텐츠입니다")
```

#### 자동 처리 규칙 (MVP)
- 동일 콘텐츠 신고 5회 누적 → 자동 숨김 (피드에서 제거, 상세 접근 불가)
- **작성자 알림**: 자동 숨김 시 작성자에게 인앱 알림 표시 ("회원님의 작품 '{제목}'이 커뮤니티 가이드라인 검토 중입니다"). 작성자는 프로필 > 알림에서 확인 가능
- 자동 숨김된 콘텐츠는 관리자(운영자)가 수동 확인 후 복원 또는 영구 삭제
- 복원 시 작성자에게 "검토가 완료되어 작품이 복원되었습니다" 알림
- MVP에서 관리자 페이지는 미구현 → Firebase Console에서 직접 확인

### 3-11. 작품 수정 / 삭제

#### 작품 수정
- 본인 작품의 작품 상세 화면 > 더보기(⋯) > "수정"
- 수정 가능 필드: 제목, 설명, 태그 (이미지 변경 불가 — MVP)
- 이미지 변경이 불가한 이유: 이미지 교체 시 Storage 삭제→재업로드 + 비정규화 thumbnailUrl 업데이트 복잡도. v2에서 지원
- **수정 저장 동작**: "저장" 탭 시 로딩 인디케이터 표시, 서버 응답 대기 후 성공 시 상세 화면으로 복귀 + "수정되었습니다" 토스트. (낙관적 업데이트 미적용 — 수정은 빈도가 낮으므로 서버 확인 후 반영)
- **수정 중 이탈 방지**: 내용이 변경된 상태에서 뒤로가기 시 discard guard 적용 ("수정 중인 내용이 있습니다. 나가시겠습니까?")

#### 작품 삭제
```
1. 본인 작품 상세 화면 > 더보기(⋯) > "삭제"
2. 확인 다이얼로그: "이 작품을 삭제하시겠습니까? 삭제된 작품은 복구할 수 없습니다."
3. 삭제 처리:
   a. Firestore artworks 문서 삭제
   b. Firestore 해당 작품의 likes 전체 삭제
   c. Firebase Storage 이미지 파일 삭제
   d. users.artworksCount -1
4. 이전 화면으로 이동 + "작품이 삭제되었습니다" 토스트
```

#### 에러 케이스
| 상황 | 처리 |
|------|------|
| 삭제 중 네트워크 끊김 | "삭제에 실패했습니다. 다시 시도해주세요" 토스트 |
| 타인의 작품에서 수정/삭제 시도 | 더보기 메뉴에 수정/삭제 항목 자체가 표시되지 않음 |
| 수정 저장 실패 | 이전 내용 유지 + "수정에 실패했습니다" 토스트 |

### 3-12. 온보딩 (첫 사용자 경험)

#### 신규 가입 직후 플로우
```
회원가입 완료
  → 온보딩 화면 1: "환영합니다! Aria에서 AI 작품을 공유해보세요"
    → 일러스트 + 간단 앱 소개 (1장 스와이프)
  → 온보딩 화면 2: "관심 있는 도구를 선택해주세요" (선택, 스킵 가능)
    → Midjourney, DALL-E, Stable Diffusion, ComfyUI 등 칩 선택 (다중 선택)
    → 선택한 도구의 인기 작품/작가를 홈 피드에 우선 노출
  → 온보딩 화면 3: "추천 작가를 팔로우해보세요" (선택, 스킵 가능)
    → 인기 작가 5명 카드 → 팔로우 버튼
  → 홈 피드로 이동
```

#### 온보딩 상태 관리
- AsyncStorage에 `onboarding_completed: true` 저장
- 앱 재시작 시 온보딩을 다시 보여주지 않음
- 온보딩 중 "건너뛰기" 탭 시 즉시 홈 피드로 이동

#### 빈 상태별 첫 사용 가이드
| 화면 | 빈 상태 UX |
|------|-----------|
| 홈 피드 (팔로우 0명) | 추천 작가 카드 3개 + "작가를 팔로우하면 여기에 작품이 표시됩니다" |
| 내 포트폴리오 (작품 0개) | 중앙 "+" 아이콘 + "첫 작품을 등록해보세요" + 업로드 화면으로 이동 |
| 좋아요한 작품 (0개) | 하트 일러스트 + "마음에 드는 작품에 ♥를 눌러보세요" + 홈 피드 이동 버튼 |
| 저장한 작품 (0개) | 북마크 일러스트 + "마음에 드는 작품을 저장해보세요" + 홈 피드 이동 버튼 |
| 방명록 (0개) | 연필 일러스트 + "아직 방명록이 없습니다. 첫 메시지를 남겨보세요!" |
| 검색 결과 (0건) | "검색 결과가 없습니다" + "다른 키워드로 검색해보세요" |
| 팔로워 목록 (0명) | "아직 팔로워가 없습니다" |
| 팔로잉 목록 (0명) | "아직 팔로우하는 작가가 없습니다" + 검색 탭 이동 버튼 |

## 4. 마이크로 인터랙션 & UX 세부 규칙

### 4-1. 피드 카드 인터랙션
| 영역 | 탭 동작 | 롱프레스 동작 |
|------|---------|---------------|
| 작품 이미지 | 작품 상세로 이동 | 없음 (MVP) |
| 작가 아바타 + 닉네임 | 작가 포트폴리오로 이동 | 없음 |
| 좋아요 하트 아이콘 | 좋아요 토글 + 바운스 | 없음 |
| 저장 북마크 아이콘 | 저장 토글 + 바운스 | 없음 |
| 카드 전체 (이미지 제외 영역) | 작품 상세로 이동 | 없음 |

### 4-2. 탭 바 인터랙션
| 동작 | 반응 |
|------|------|
| 다른 탭으로 이동 | 크로스페이드 전환 |
| **현재 탭 아이콘 재탭** | **해당 탭의 리스트가 최상단으로 스크롤** (Instagram 패턴) |
| 현재 탭 아이콘 재탭 (스택이 쌓인 상태) | 먼저 루트 화면으로 pop, 이미 루트이면 최상단 스크롤 |

### 4-3. 햅틱 피드백
| 동작 | 햅틱 종류 |
|------|-----------|
| 좋아요 토글 | `impactLight` |
| 저장 토글 | `impactLight` |
| 팔로우 토글 | `impactLight` |
| Pull-to-refresh 트리거 | `impactMedium` |
| 등록 완료 | `notificationSuccess` |
| 에러 발생 (토스트) | `notificationError` |
| 작품 삭제 확인 | `notificationWarning` |
| 이미지 드래그 순서 변경 | `selectionChanged` (드래그 중 각 위치마다) |

> **초보자 참고** — 햅틱 피드백은 사용자가 탭하거나 동작을 수행할 때 디바이스가 짧게 진동하는 것입니다.
> Expo의 `expo-haptics` 라이브러리로 간단히 구현합니다. Android에서는 기기에 따라 지원 여부가 다릅니다.

### 4-4. 키보드 처리
| 화면 | 키보드 동작 |
|------|-------------|
| 로그인/회원가입 | `KeyboardAvoidingView`로 입력 필드가 키보드에 가려지지 않음. 키보드 위 "완료" 버튼 |
| 작품 등록 (제목/설명) | 스크롤 가능한 폼. 활성 입력 필드 위치로 자동 스크롤 |
| 방명록 입력 | 입력 바가 키보드 위에 고정 (채팅 패턴). 키보드 올라올 때 메시지 목록도 위로 이동 |
| 검색 | 검색 바 포커스 시 키보드 올라옴. 결과 탭 시 키보드 자동 내림 |
| 프로필 수정 | `KeyboardAvoidingView` + 스크롤. 키보드 밖 영역 탭 시 키보드 내림 |
| 공통 | 키보드 밖 영역 탭 = `Keyboard.dismiss()` (검색 제외: 검색은 결과를 탭해야 내림) |

| 회원가입 (중간 단계) | 뒤로가기 | 가드 없음 (회원가입 재시작 가능) |

### 4-10. 스크롤 위치 복원
| 상황 | 동작 |
|------|------|
| 피드에서 작품 상세 진입 → 뒤로가기 | 피드의 이전 스크롤 위치 복원 |
| 포트폴리오 그리드에서 작품 상세 진입 → 뒤로가기 | 그리드의 이전 스크롤 위치 복원 |
| 검색 결과에서 작품/작가 진입 → 뒤로가기 | 검색 결과의 이전 스크롤 위치 복원 + 검색어 유지 |
| 탭 전환 후 복귀 | 각 탭의 스크롤 위치 유지 |
| Pull-to-refresh 후 | 최상단으로 스크롤 (새 데이터가 상단에 추가되므로) |

### 4-11. 네비게이션 스택 깊이 제한
```
피드 → 작품A → 작가X → 작품B → 작가Y → 작품C → ...

스택이 무한히 깊어지는 것을 방지:
- 제한하지 않음 (React Navigation 기본 동작)
- 단, 동일 화면 중복 진입 방지:
  - 현재 보고 있는 작가 포트폴리오에서 같은 작가의 다른 작품 → 작품 상세 진입 → OK
  - 작품 상세에서 같은 작품의 작가 미니 카드 탭 → 이미 해당 작가 포트폴리오가 스택에 있으면 popToTop 후 포트폴리오로 이동
```

## 5. 비기능 요구사항

### 5-1. 성능
| 지표 | 목표 |
|------|------|
| 앱 시작 → 홈 피드 최초 렌더 | 3초 이내 (네트워크 양호 시) |
| 작품 이미지 로딩 (피드 썸네일) | 1초 이내 |
| 좋아요/팔로우 UI 반응 | 즉시 (낙관적 업데이트) |
| 검색 결과 표시 | 2초 이내 |
| 작품 등록 (이미지 1장, 2MB) | 5초 이내 |

### 5-2. 오프라인 동작
| 상황 | 동작 |
|------|------|
| 앱 시작 시 오프라인 | 캐시된 피드 표시 + 상단 배너 "오프라인 상태입니다" |
| 사용 중 네트워크 끊김 | 진행 중 작업 중단, 현재 화면 유지, 배너 표시 |
| 네트워크 복원 | 배너 "다시 연결되었습니다" (초록색, 2초 후 자동 제거) → 현재 화면 데이터 자동 갱신 (silent refetch) |
| 복원 중 진행 중이던 쓰기 작업 | 자동 재시도하지 않음. "연결이 끊겨 저장되지 않았습니다. 다시 시도해주세요" 토스트 표시 |
| 오프라인에서 좋아요/팔로우 시도 | "인터넷 연결이 필요합니다" 토스트 |
| 오프라인에서 작품 등록 시도 | 업로드 버튼 비활성화 + 안내 메시지 |

### 5-3. 접근성
- 모든 터치 타겟 최소 44x44px
- 이미지에 alt 텍스트 제공 (작품 제목 사용)
- 색상 대비 WCAG AA 기준 (4.5:1) 이상
- 스크린 리더 지원: 주요 버튼과 내비게이션에 accessibilityLabel 제공

### 5-4. 보안
- Firebase Security Rules로 서버사이드 데이터 접근 제어
- 사용자 입력 XSS 방지: React Native는 기본적으로 HTML 렌더링하지 않으므로 XSS 위험 낮음. 단, WebView 사용 시 주의
- 이미지 업로드 시 Content-Type 검증 (서버사이드)
- Rate Limiting: Firebase Auth 기본 제공 (IP당 로그인 시도 제한)
- **이미지 업로드 Rate Limiting**: Security Rules에서 사용자당 1분 내 5회 이상 업로드 차단. 클라이언트에서도 연속 업로드 시 "잠시 후 다시 시도해주세요" 안내
- 민감 정보(비밀번호 등)는 로그에 출력하지 않음
- **닉네임 유일성**: Firestore Transaction으로 닉네임 중복 체크 + 저장을 원자적으로 처리. `nicknames` 컬렉션에 normalizedNickname을 문서 ID로 저장하여 유일성 보장 (race condition 방지)
- **방명록 스팸 방지**: 클라이언트에서 전송 버튼 쿨다운(3초) 적용. 서버사이드 rate limiting은 v2에서 Cloud Functions로 강화 예정
- **고아 Storage 파일 정리**: 작품 삭제 시 서비스 레이어에서 Storage 이미지도 함께 삭제. 업로드 후 Firestore 문서 생성 실패 시 업로드된 이미지 롤백 삭제. MVP에서 미처리된 고아 파일은 Firebase Console에서 수동 정리

### 5-5. 개인정보
- 수집 항목: 이메일, 닉네임, 프로필 사진, 작품 이미지

> **초보자 참고 — Firebase란?**
> Firebase는 Google이 만든 "백엔드 세트 도구"입니다.
> 서버를 직접 만들지 않아도 로그인, 데이터베이스, 파일 저장을 앱에서 바로 사용할 수 있습니다.
> 무료 Spark 플랜으로 소규모 서비스 운영이 충분히 가능합니다.
> Spark 플랜 한도: Firestore 읽기 50K/일, 쓰기 20K/일, Storage 5GB, Auth 무제한.

### 비정규화 전략 — 왜 데이터를 중복 저장하는가?

> **초보자 참고**
> Firestore는 SQL 데이터베이스의 JOIN이 없습니다.
> "작품 카드에 작가 닉네임을 표시"하려면 artworks 문서를 읽은 뒤 users 문서를 또 읽어야 합니다.
> 피드에 20개 카드가 있으면 20번 추가 읽기가 발생 → 느리고 비용도 높습니다.
>
> 그래서 artworks 문서에 authorNickname을 미리 복사해둡니다.
> 단점: 닉네임이 변경되면 모든 작품의 authorNickname도 업데이트해야 합니다.
> 이 작업은 Cloud Function (또는 일괄 업데이트 스크립트)로 처리합니다.

| 비정규화 필드 | 원본 위치 | 동기화 시점 |
|---------------|-----------|-------------|
| artworks.authorNickname | users.nickname | 닉네임 변경 시 batch update |
| artworks.authorProfileImageUrl | users.profileImageUrl | 프로필 사진 변경 시 batch update |
| guestbooks.authorNickname | users.nickname | 닉네임 변경 시 batch update |
| users.followersCount | follows 컬렉션 count | 팔로우/언팔로우 트랜잭션 |
| users.artworksCount | artworks 컬렉션 count | 작품 등록/삭제 트랜잭션 |
| artworks.likesCount | likes 컬렉션 count | 좋아요/취소 트랜잭션 |
| users.bookmarksCount | bookmarks 컬렉션 count | 저장/취소 트랜잭션 |

### Firestore 인덱스 (필수)

| 컬렉션 | 필드 | 용도 |
|--------|------|------|
| artworks | isHidden ASC, createdAt DESC | 홈 피드 (숨겨지지 않은 작품, 최신순) |
| artworks | authorId ASC, createdAt DESC | 작가 포트폴리오 (특정 작가 작품, 최신순) |
| artworks | isHidden ASC, likesCount DESC | 인기 작품 정렬 |
| likes | userId ASC, createdAt DESC | 좋아요한 작품 목록 |
| follows | followerId ASC, createdAt DESC | 내 팔로잉 목록 |
| follows | followingId ASC, createdAt DESC | 내 팔로워 목록 |
| bookmarks | userId ASC, createdAt DESC | 저장한 작품 목록 |
| reports | targetId ASC, reporterId ASC | 중복 신고 방지 |
